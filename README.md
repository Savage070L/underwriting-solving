# Underwriting App — Standard Life

Генератор андеррайтинговых документов и аналитический дашборд по обязательному страхованию работников от несчастных случаев.

## Возможности

- Загрузка справочников (поправочные коэффициенты, норматив, КУ по классам, калькулятор рентабельности) с кешированием в `localStorage`
- Парсинг заявки на андеррайтинг (`.xlsm`) и истории убытков (`.xls`)
- Автоматический поиск адреса и доли государственного участия по БИН (через Cloudflare Worker)
- Генерация в `.docx`:
  - **Андеррайтинговое решение (АР)**
  - **Заключение департамента андеррайтинга**
  - **Протокол** заседания Андеррайтингового Совета / Правления
- Полноценный аналитический дашборд (`analytics.html`) с 40+ карточками: KPI, Risk Score, тяжесть, страховщики, актуарные показатели (Burning Cost, Pure Premium, PML, RAROC), стресс-тесты, рекомендации по перестрахованию

## Запуск локально

```bash
cd underwriting-app
python3 -m http.server 8000
# открыть http://localhost:8000
```

или любой другой статический HTTP-сервер.

## Деплой на GitHub Pages

1. Загрузить эту папку в репозиторий GitHub
2. В Settings → Pages выбрать source = «GitHub Actions»
3. Push в `main` → workflow [`.github/workflows/pages.yml`](.github/workflows/pages.yml) автоматически развернёт сайт
4. URL будет вида `https://<user>.github.io/<repo>/`

## BIN-поиск и CORS

Поиск по БИН (адрес из `pk.uchet.kz`, доля гос. участия из `e-Qazyna`) **не работает напрямую с GitHub Pages** из-за CORS-ограничений браузера. Решение — Cloudflare Worker-прокси:

- Папка [`worker/`](worker/) содержит готовый Worker (`index.js`) и `wrangler.toml`
- Деплой: `cd worker && wrangler deploy`
- В [`js/app.js`](js/app.js) обновить константу `App.WORKER_URL` на ваш URL после деплоя

## Признак резидентства (ГБД ЮЛ)

Резидентство определяется по БИН: **есть в реестре юр. лиц РК ([gbd_ul](https://data.egov.kz/datasets/view?index=gbd_ul)) → резидент, нет → нерезидент.** Запросов на каждый БИН нет — весь реестр (757 231 БИН) лежит рядом с приложением в виде бинарного индекса `data/gbd_ul_bins.bin` (~0,8 МБ), грузится один раз и дальше проверка мгновенная и офлайн.

### Автообновление индекса (GitHub Actions)

Индекс пересобирается **автоматически раз в неделю** — [`.github/workflows/refresh-gbd-index.yml`](.github/workflows/refresh-gbd-index.yml), воскресенье ~20:30 по Алматы. Workflow сам выкачивает свежий реестр, коммитит новый `data/gbd_ul_bins.bin`, передеплоивает Pages и шлёт письмо об итоге (успех/ошибка). При неполной выгрузке индекс **не** перезаписывается — старый остаётся рабочим. Запустить вручную: вкладка **Actions → «Обновление индекса резидентства (ГБД ЮЛ)» → Run workflow**.

**Одноразовая настройка почты** (без неё письма не уйдут, всё остальное работает):

1. В Google-аккаунте включите двухфакторную аутентификацию и создайте **пароль приложения** (App Password): <https://myaccount.google.com/apppasswords>. Это 16-значный пароль, обычный пароль Gmail для SMTP не годится.
2. В репозитории: **Settings → Secrets and variables → Actions → New repository secret**, добавьте два секрета:
   - `MAIL_USERNAME` — ваш адрес Gmail (на него же придёт письмо);
   - `MAIL_PASSWORD` — тот самый пароль приложения.
3. Прогоните workflow вручную (Run workflow) — проверить, что письмо доходит.

Каждый недельный коммит индекса ~0,77 МБ (история растёт ~40 МБ/год — при желании раз в пару лет чистится `git gc`/squash).

### Обновление вручную

```bash
python3 tools/build_gbd_index.py --refresh
```

Дата сборки видна в интерфейсе («ГБД ЮЛ от ДД.ММ.ГГГГ»). ИИН (ИП и физлица) в реестр юр. лиц не входят — такие строки помечаются «ИП» и по умолчанию считаются резидентами.

Если выгрузка оборвалась (egov на длинных обходах иногда перестаёт отвечать) — просто запустите ту же команду ещё раз: прогресс пишется в `data/.gbd_ul_scan.tmp`, и скрипт продолжит с места обрыва. Начать заново — флаг `--restart`.

Три особенности выгрузки, которые учтены в скрипте:

- **Обход реестра идёт курсором** (`range: bin > последний`, `sort: bin asc`, `from` всегда 0). Обычный обход через `from`/`size` на глубоких смещениях теряет записи — на реальном прогоне пропало 655 действующих компаний из 753 520, и они стали бы «нерезидентами». В конце скрипт сверяет число прочитанных записей с `totalCount`.
- **По одному БИН в реестре бывает несколько записей** (перерегистрация: старое и текущее наименование/адрес). Берётся **самая свежая** — по дате регистрации, при равных датах по актуальности загрузки. Её статус («Ликвидирован» / «Реорганизован») попадает в индекс отдельным флагом: резидентства это не отменяет, но в интерфейсе показывается жёлтой пометкой «резидент · ликвидирован».
- **Записи с пустым БИН пропускаются** — в наборе их ~50 тыс., и это дубликаты тех же компаний, у которых БИН заполнен.

## Почему НЕ WebAssembly

- Парсинг Excel: библиотека [`xlsx.full.min.js`](js/lib/xlsx.full.min.js) — pure JavaScript, работает быстро в любом современном браузере
- Генерация .docx: [`docx.min.js`](js/lib/docx.min.js) — также pure JS
- BIN-поиск: ограничен CORS, не размером данных — WASM не решает эту задачу
- Аналитика (Compound Poisson, регрессии, Pareto, HHI): тысячи операций на стороне клиента — JS справляется за миллисекунды

Если в будущем понадобится тяжёлая числовая обработка (например, Monte Carlo симуляция 100 000 итераций или сложные актуарные методы), можно интегрировать WASM-модуль, но в текущем сценарии — оверкилл.
